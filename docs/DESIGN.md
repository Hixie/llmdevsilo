# llmdevsilo design

Premise: Existing LLMs/agents are powerful tools for coding, both in terms of potential productivity and in terms of potential damage. We need to create an agent harness that leverages LLMs and modern sandboxing technologies to create a *safe* environment that achieves similar productivity gains.

* To achieve the productivity, LLMs need:
  * access to relevant tools on your machine, e.g. runtimes, command line tools like git, grep, python, etc, the ability to execute arbitrary code, etc.
  * access to all the source code that you are working on.
  * the ability to run the source code, its tests, and other development tools (compiler, code analyzers).
  * access to the Internet (to access supporting documentation, do research, test clients being developed against real dev servers, etc).
  * the ability to run subagents.
  * ability to install and run new programs (within the sandbox).
  * ability to run native code (e.g. macOS programs on macs, Linux programs on Linux).
* The attack vectors (how the model can be misaligned with the user) are:
  * prompt injection in third-party documentation: the model reads content from a dependency, or information on a web page like reddit or StackOverflow, that misguides it towards goals the user doesn't want.
  * latent triggers in the model itself: the model was trained on tainted data that has left an undetected, undocumented, or just unexpected trigger that will lead it to behave against the user's goals in certain circumstances.
  * misalignment in the model training: the model was trained, intentionally or not, to pursue goals that are at odds with the user's.
  * mistakes in user input: the user can give guidance that is ambiguous or otherwise doesn't reflect the user's intent (e.g. a typo changing "not" to "now").
  * permission request fatigue: human users quickly tire of checking for potential risks and therefore permission requests are not a viable security policy.
* The attack payloads (potential bad effects that these attack vectors might trigger) that we want to statically mitigate are:
  * Permanent damage to the user's file system, e.g. deleting the home directory.
  * Permanent hardware damage outside of normal wear and tear (e.g. triggering a battery fire is in scope; worsening battery performance due to heavy CPU use is not).
  * Privilege escalation, e.g. wherein the LLM gains some measure of control over the kernel that the user did not expect.
  * Any form of user-mediated sandbox escape, e.g. the LLM poisoning the environment with git hooks, IDE configuration files, .envrc, etc, that the user may not expect. (This does not include the code itself being malicious.)
  * Exfiltration of the user's personal data or credentials, including credentials held by the harness itself such as API keys and frontend keys.
  * Exfiltration of authentication tokens used by the LLMs to achieve certain tasks like reading bug databases or committing code to a source code versioning repository.
  * Exfiltration of any data managed by other groups of LLMs (i.e. other harnesses, in this proposal).
  * Exfiltration of any part of an LLM's conversation with the user.
  * Exfiltration of any cookies or other credentials stored by the user's web browser.
  * Exfiltration of intranet-accessible data (fetching data that is only accessible due to the user's ambient authority granted by their network, and then sending it elsewhere).
* There are some acceptable risks, attacks that we do not want to mitigate or that we consider out of scope due to there not being any reasonable way to mitigate them:
  * Adding misaligned code to the codebase. Since the goal is to have the LLMs write code, and since aligned code and misaligned code cannot fundamentally be distinguished statically, we can only rely on human and LLM review to detect misaligned code. LLMs are unlikely to reliably be able to catch intentionally misaligned code, as they will likely be susceptible to the same attack vector (e.g. prompt injection) that led to the original misalignment. Humans are unlikely to be able to reliably catch intentionally misaligned code due to the difficulty of detecting intentionally hostile code, review fatigue, and confirmation bias. *Unintentional* misalignment (bugs introduced by the model, or error caused by the original user misprompting) is more likely to be caught, but is out of scope of this proposal as well.
  * Vandalizing uncommitted code (e.g. the model deleting everything it has write access to). Since there is fundamentally no difference between the model deleting old code or temporary files, and the model deleting all code and files, this is a lost cause. The damage is scoped to that LLM group's output, however, and therefore this damage is minimal in impact. A versioning filesystem could mitigate this in principle but it is not expected to be a common problem.
  * Temporarily vandalizing the host computer, e.g. causing a kernel panic, overheating the CPU, using all memory, using all disk. While temporarily disruptive, and potentially causing short-term damage (e.g. loss of newly created data), these attacks by definition are temporary and thus of little interest to attackers, and they are typically easy to mitigate once experienced (avoid the trigger, put in place measures to detect the specific attack).
  * Exfiltration of any source code or data the model has been intentionally given access to. We assume that all code being developed is or will be open source and therefore there are no secrets accessible to the LLM. We similarly assume that any credentials accessible to the LLM are temporary development credentials. We assume that production data and environments are not exposed to the LLM. This must be stated up front in any documentation.
  * Exfiltration of binaries, libraries, or supporting data files that ship with the operating system (e.g. `/usr/bin`) and any tools installed by the user (e.g. git, compilers) that are not unique to the installation or to the user.
  * Exfiltration of any system-specific (not user-specific) identifiers like MAC addresses, SSH public host keys, disk UUIDs. Where these can be anonymized, this must be done, but this is considered best-effort.
  * Exfiltration to the cloud LLM providers of any workspace or conversation content. The cloud LLM providers are assumed to be trustworthy.
  * Kernel security bugs. We can assume that the kernel implements its APIs as documented, unless there are known unmitigated attack vectors. Where possible, kernels must be insulated from code running in the sandbox, but this is considered best-effort.
* Design:
  * The core deliverable is a harness that coordinates LLM backends, with a sandbox and workspace, and a frontend. It is a Rust binary that runs on macOS or Linux workstations, and on Linux servers. Windows is considered out of scope for now.
  * Frontend:
    * The harness must have a modular frontend architecture. The default Rust deliverable must ship with, at a minimum: a WebSocket server that user applications can connect to known as the *interactive frontend*, a *headless frontend* for running in the background, and a *mock UI frontend* for tests (which sends messages to the harness following a script and checks that responses are as expected). Each instance of a harness has one such frontend selected.
    * The interactive frontend module must require authentication for clients to connect. For local clients, a key shared via the local filesystem can be used for seamless authentication. For remote clients, the harness should have an affordance for providing a one-time short-lived pairing code, which the client can send to confirm its identity; the client should then immediately generate an asymmetric key pair and provide the harness with the public key. Subsequent connections can be made by having the client sign a harness-issued challenge. The client should store the private key using platform-native best practices.
    * The interactive frontend must support multiple simultaneous connections from multiple clients. The protocol must support keeping all clients synchronized (e.g. a prompt sent by one client should be displayed on all clients; a file uploaded in one client should be visible in all clients; any responses from the LLM must be sent to all clients; etc). When an LLM sends a question (AskUserQuestion), the question is shown on all connected interactive frontends and the answer comes from the first to respond (which then removes the question UI from the other frontends, showing only the question/answer in the conversation flow). Similarly, when an LLM is waiting for user input, the first frontend to send input starts the LLM's next turn.
    * Events should be identified with an incrementing counter starting at zero. Connecting clients should be able to request all events from a particular counter.
    * The headless frontend accepts a prompt on the command line, sends that to the LLM as the first turn along with an instruction to use an "Exit" tool when the task is complete, then responds to all subsequent LLM requests with "This is a non-interactive session; complete the task then use the Exit tool". The headless frontend adds an "Exit" tool to the LLM backend for top-level agents, and terminates when the LLM calls that tool with a message, presenting that message to the user.
    * The interactive frontend adds "AskUserQuestion" and "SendUserFile" to the top-level agent.
  * LLM backends:
    * There must be multiple LLM backends, including at a minimum the Anthropic Messages API REST backend, the OpenAI Responses REST backend, the OpenAI WebSocket backend (without audio), a mock backend for testing, and a local model backend that runs a model locally.
    * The LLM backends must support being configured with tools by the other components of the harness. In addition to the schema that the tool must follow, the configuration should also include whether the tool is available to top-level agents, subagents, or both.
    * The LLM backend must add the following tool: Agent (to launch a subagent).
    * The LLM module may communicate with a remote LLM in the cloud if necessary (a locally hosted LLM would not require this, and the mock LLM does not require this). All other communication must be via the harness.
    * LLM backends that use models with usage costs (e.g. using Anthropic or OpenAI servers) must have quota systems to prevent runaway usage, and must have an API that reports ongoing cost in tokens and dollars, which must be exposed in the frontends.
  * Workspace sandboxing (managed by the harness):
    * The user must be able to pick a directory and specify that it is a workspace for a harness' sandbox. This must be considered *locking* the workspace.
    * When launching the harness, a locked workspace must be specified (or a new empty locked workspace can be created on the fly); the sandbox is given this workspace.
    * When the workspace is locked, the user must not be able to interact with the files in the workspace from the unsandboxed environment.
    * The user must be able to unlock a workspace environment. This must prevent the LLM from interacting with it. When unlocking a workspace environment, any sandbox using that workspace must be forcibly terminated, ending any associated harness, and the files that have in any way changed since the last time the environment was locked must be listed with a diff from their state when the workspace was locked. As an additional layer to help the user review the changes, any changes to known auto-exec surfaces (including but not limited to git hooks, `.envrc`, IDE configurations such as VSCode's `.vscode/tasks.json`, package manager scripts such as npm's `package.json postinstall`, etc) must be flagged explicitly.
    * Locking/unlocking can be implemented by mounting a file as a loop device inside the sandbox; this may vary as necessary per platform. On platforms where the capability to mount a loop device exclusively inside the sandbox is not present, the user should be warned about the risks of accessing the contents from outside the sandbox while it is locked.
  * Sandbox backends:
    * The sandboxing technology must be modular, with different options provided to the user. On Linux, gVisor must be one option, and a microVM hypervisor should be another (e.g. based on Firecracker). On macOS, sandbox-exec must be one option (it's deprecated but really the only option for developers wanting to create macOS programs); and a Linux VM via Virtualization.framework must be another. (These are not interchangeable options; they address different use cases, and have different security tradeoffs.) There must also be a mock version that verifies that tool requests are as expected and that plays back responses according to a script.
    * The sandbox has a filesystem known as a workspace that is read/write, but not accessible to the user. In addition it must have read access to a user-configured allow-listed subset of the host filesystem, e.g. `/usr/bin`. The inability for the sandbox to read sensitive data prevents that data from being exfiltrated. The inability for the sandbox to write outside the workspace prevents catastrophic vandalism.
    * In addition, the sandbox must have a per-sandbox writable scratch space to which toolchains can write temporary files. This must be cleaned up on exit and must not be accessible from outside the sandbox, like a locked workspace.
    * For gVisor-based and Linux-VM-based sandboxes: The sandbox must funnel all *egress* to a harness-controlled transparent proxy, with the sandboxing routing forcing all traffic through this proxy. This proxy must enforce a domain allowlist, block the host's intranet, localhost, and link-local ranges (including *after* DNS resolution, both IPv4, IPv6, and IPv4-over-IPv6), terminate all TLS to/from the sandbox with a custom CA (per-session, ephemeral, never-persisted certificate authority; with only the public key readable from the sandbox), and log traffic. For testing purposes, this proxy must be mockable. It must support injecting credentials for specific domains (e.g. if an LLM is configured to have GitHub access, those credentials must be injected by the proxy rather than accessible directly to the LLM). Anything that cannot be proxied must be blocked. DNS specifically must be proxied and only allowlisted domains must be resolvable. The inability to reach the local network prevents local network data from being exfiltrated. The credential injection allows for authenticated access to resources without leaking credentials.
    * For macOS sandbox-exec sandboxes: All outbound network traffic is restricted, and one local port is used for an HTTP proxy implemented by the sandbox component (internal loopback traffic is unaffected). All outbound traffic in the sandbox is routed to that proxy. The proxy implements the credential injection, logging, etc, described above. (This blocks direct network connections, but that is unavoidable.)
    * The sandbox must offer an API that lists what an LLM can access, e.g. a directory listing of all the files it can read or execute outside its workspace, any allowlisted domains, etc. (The interactive frontend should expose this API to its clients, and the GUI applications should make use of this to allow for an at-a-glance view of the sandbox's access.)
    * Within the sandboxed environment, a process must connect to the sandbox module to be told what commands to run. Specifically, there is a component running within each sandbox, as untrusted code, that can receive messages from the harness and respond to them.
    * Programs executed within the sandbox must have all capabilities that would enable any of the attack vectors described earlier filtered. For example, it must not be possible for such programs to mount or unmount filesystems outside the sandbox, change the system time, bring up network interfaces, etc.
    * The sandbox must add tools to the LLM backend: Read/Write/Edit, Bash, WebFetch, WebSearch. These must be implemented by the sandboxed helper process, so they are subject to all the sandbox restrictions.
    * The user must be able to launch into an environment that sandboxes the user in a manner identical to how the LLM's work is sandboxed, allowing the user to run the code, explore it, edit it, etc. It must be possible to launch interactive tools (e.g. VSCode) from this environment (leaving them similarly sandboxed, and avoiding any security issues from auto-exec surfaces, as those would be sandboxed and thus unable to perform any actions not already possible by the model directly).
  * Harness:
    * The harness coordinates a sandbox, an LLM and any of its subagents, and a frontend.
    * The harness runs until the frontend requests that the harness shut down, or until the harness receives an appropriate signal.
    * There is one LLM originally, but more can be spawned by that LLM using a subagent tool. All LLMs in a harness share the same sandbox. It is the LLM's responsibility to ensure their work does not conflict (e.g. two subagents attempting to modify the same file in conflicting ways). (This is similar to how Claude for Desktop works.)
    * Tool executions by the LLMs, and any communication between agents and subagents, are forwarded to the frontend to be surfaced in the UI in some way.
    * The harness must persistently log activity between its various modules, including all interactive commands, all LLM input and output, any activity triggered by the LLM (e.g. every executed command), and most network traffic. The logging infrastructure must be verbose enough that a log file can be used to generate a test that reproduces the same interactions using mock components. (This must be at the level of LLM tool interactions, not syscalls; in tests, the sandboxed environment must not actually execute code or store files.) Log must be designed to only carry non-sensitive data (e.g. the actual credentials injected by the proxy part of the sandbox would not be logged, as they can be recreated during replay and are not needed for debugging). Logs must not be accessible from the sandbox.
  * Interactive frontends (independent applications, not part of the core harness):
    * There must be several independent applications, including: interactive beautiful native apps (desktop and mobile), an interactive colorful command line tool, and an interactive HTML5-style web interface. They must communicate with the interactive frontend WebSocket server of the harness over a TLS-encrypted WebSocket, allowing the harness to run on a different host than the UI, e.g. allowing for a mobile interface to connect to the harness or for harnesses to be spawned across multiple servers and controlled from one computer. Multiple UIs must be able to simultaneously connect to the same harness, and show all remain synchronized.
    * The native GUI app and the web interface must support connecting to multiple harnesses at a time. Clients that support this, on platforms that support it, must be able to spawn new local harnesses to connect to. (Spawning server-hosted harnesses from the GUI applications is out of scope, as is spawning local harnesses on web or mobile.)
  * Threat analysis:
    * Permission request fatigue: the design features no permission prompts; however, user review of model output remains a weak point.
    * Prompt injection in third-party documentation, latent triggers in the model itself, misalignment in the model training, and mistakes in user input all manifest as the LLMs attempting to execute misaligned actions intended to result in the aforementioned damages, which are discussed forthwith.
    * Permanent damage to the user's file system: avoided because the sandbox cannot access the sensitive parts of the user's filesystem.
    * Permanent hardware damage and automated privilege escalation: prevented by blocking access to the hardware (no root/sudo access, minimal kernel access).
    * Exfiltration of the user's personal data or credentials, including any cookies or other credentials stored by the user's web browser: blocked by the sandbox not having access to that data.
    * Exfiltration of authentication tokens used by the LLMs: where possible, blocked by the injected credentials model; this is, however, a potential weak point as some services may not support that.
    * Exfiltration of any data managed by other groups of LLMs, or of an LLM's conversation with the user: blocked by the sandbox as the data managed by those models, and the logs, are outside the sandbox allowlist.
    * Exfiltration of intranet-accessible data: blocked by preventing any ingress of intranet content to the sandbox.
    * User-mediated sandbox escape: this is the least secure aspect of the design, relying on user review of model work.
    * Blocking network egress is only required to prevent fetching intranet resources. It is not a load-bearing part of the exfiltration mitigation; for exfiltration, it serves only as a defense-in-depth. Thus, for instance, providing GitHub access to the sandbox does not risk exfiltration. Similarly, there is no additional risk posed by content being smuggled through URLs or DNS requests.
  * Design limitations:
    * The user is expected to be responsible for managing what credentials are provided for network calls (e.g. using scoped tokens that only provide read access to sensitive repositories).
    * The MITM approach to proxying TLS data will break certificate-pinning package managers. This will be revisited once the scope is better understood by testing with an implementation. Eventually maybe the user will be able to whitelist specific hosts (IPs) to have direct access out of the sandbox.
    * Tests involving log data must be designed carefully to avoid introducing potential race conditions (e.g. if an LLM is faced with periodic messages coming from both the sandbox and the frontend, in principle these could race). This must be achieved by having events have sequence numbers and by having the harness manage a fake clock and having all mock components follow either sequence numbers or that fake clock to trigger events, rather than using independent timers.
    * Because the user configures the read allowlist, it's possible to accidentally add sensitive data to it (e.g. adding one's home directory exposes one's SSH keys). To mitigate this, the harness must ship with a hardcoded list of known risks, and must scan for these and block the user from adding them accidentally. This is a best-effort defense-in-depth to minimize accidental risk and not a core part of the security model.
  * Design choices:
    * The core language (used for the harness, its components, and the command line frontend application) will be Rust.
    * The interactive desktop, web, and mobile frontend will be a Flutter application.
